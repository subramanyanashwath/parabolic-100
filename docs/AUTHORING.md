# Authoring a worked example

Every problem earns its slot as the cheapest representative of a recurring
situation class (see COVERAGE.md). The bar:

1. **The prompt is a situation, not a topic.** Someone wants something,
   something is broken, and a decision is due — with names, numbers, and
   constraints. If it reads like a textbook section heading, start over.
2. **The expected falsifier could flip the call.** "Measure carefully" is not
   a falsifier. State the evidence that would prove the model answer wrong.
3. **The adversarial critique attacks smart answers.** Including the
   over-corrected one. If it only demolishes strawmen, it trains nothing.
4. **Altitude rewrites change framing, never facts.** Exec, engineer, and
   interviewer hear the same judgment at three altitudes.
5. **Tag or be rejected:** category, 1–3 patterns, situation class, format
   (case / design / behavioral / exec-comm), and concept tags where the
   problem presumes technical mechanisms.
6. **New patterns need three problems.** A pattern enters the taxonomy only
   when three problems require it and no existing pattern fits.
7. **Fictional composites only.** Public knowledge, no proprietary
   information, no real customer situations, no employer internals.

Schema reference: see any entry in `problems-1.js`. Match its depth — the
model answer argues, the rubric descriptors are per-problem, and the
recursive follow-up makes the situation one turn harder.
